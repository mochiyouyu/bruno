# Bruno 场景与场景套件使用说明

## 1. 功能概述

Runner 现在支持两层复用能力：

- 场景 `Scenario`：保存一组接口、执行顺序、延迟和标签过滤条件，方便重复执行。
- 场景套件 `Scenario Suite`：把多个已保存的场景按顺序组合起来，形成一条自动化执行链路。

适合的典型用法：

- 用一个场景表示一段固定测试步骤，比如“登录冒烟”“下单主流程”“回归校验”。
- 用一个套件把多个场景串起来，形成“每日巡检”“提测验证”“回归全链路”等自动测试任务。

## 2. 入口位置

1. 打开一个 Collection。
2. 点击顶部的 `Runner`。
3. 左侧可以看到 `Scenario` 和 `Scenario Suite` 两个区域。

## 3. 场景会保存什么

每个场景会保存以下内容：

- 场景名称
- 已勾选的请求列表
- 请求执行顺序
- 请求间隔 `Delay between requests (ms)`
- 标签过滤条件 `Include tags / Exclude tags`

## 4. 如何创建场景

1. 在 Runner 右侧请求列表中勾选要执行的接口。
2. 通过拖拽右侧列表调整执行顺序。
3. 按需设置：
   - `Delay between requests (ms)`
   - `Include tags`
   - `Exclude tags`
4. 点击 `Scenario` 区域中的 `Save As`。
5. 输入场景名称并保存。

保存后，该场景会出现在场景下拉框中，后续可以直接复用。

## 5. 如何修改场景

### 5.1 更新当前场景

适合保留场景名称，只更新内容：

1. 从 `Scenario` 下拉框中选择目标场景。
2. 修改请求勾选、执行顺序、延迟或标签。
3. 点击 `Update`。

### 5.2 另存为新场景

适合基于已有场景派生一份新版本：

1. 选择一个已有场景。
2. 调整请求或配置。
3. 点击 `Save As`。
4. 输入新的场景名称并保存。

## 6. 如何执行单个场景

1. 从 `Scenario` 下拉框中选择目标场景。
2. Runner 会自动恢复该场景保存时的：
   - 已选请求
   - 请求顺序
   - Delay
   - 标签过滤条件
3. 点击 `Run`。

## 7. 如何创建场景套件

场景套件用于把多个已保存的场景组合成一次自动化测试。

1. 先确保已经保存了两个或以上场景。
2. 在 `Scenario Suite` 区域中勾选要纳入套件的场景。
3. 使用每个场景右侧的上移、下移按钮调整执行顺序。
4. 点击 `Save As`。
5. 输入套件名称并保存。

套件会保存：

- 套件名称
- 参与执行的场景列表
- 场景执行顺序

## 8. 如何修改场景套件

1. 从 `Scenario Suite` 下拉框中选择目标套件。
2. 勾选或取消勾选场景。
3. 调整场景顺序。
4. 点击 `Update`。

如果想保留原套件并派生新版本，可以直接点击 `Save As`。

## 9. 如何执行场景套件

1. 在 `Scenario Suite` 下拉框中选择一个套件，或者临时勾选多个场景形成一个未保存套件。
2. 点击 `Run Suite`。
3. 系统会按套件中的场景顺序依次执行。
4. 每个场景内部仍按各自保存的请求顺序执行。

执行结果页会额外展示：

- 当前套件名称
- 每条请求所属的场景名称

## 9.1 自动化测试的推荐使用流程

如果要把这个能力当成日常自动化测试入口来使用，建议按下面的方式组织：

1. 先按业务链路拆分多个场景。
2. 每个场景只保存一段相对稳定、可复用的步骤。
3. 再把多个场景组合成一个套件。
4. 使用套件执行整套自动化测试。

推荐的拆分方式示例：

- `Smoke - 登录与健康检查`
- `Core - 下单主流程`
- `Verify - 数据校验与清理`
- `Nightly Regression`

这样做的好处是：

- 单个场景便于独立调试
- 套件便于组合执行
- 后续扩展时只需要替换某个场景，不必重建整套流程

## 9.2 一次完整自动化测试的操作步骤

下面是一套从零开始的完整使用方式：

1. 打开目标 Collection。
2. 进入 `Runner`。
3. 配置并保存第一个场景，例如“冒烟检查”。
4. 配置并保存第二个场景，例如“主流程回归”。
5. 如有需要，继续保存第三个、第四个场景。
6. 在 `Scenario Suite` 区域勾选这些场景。
7. 用上移、下移按钮排好执行顺序。
8. 点击 `Save As` 保存为一个套件。
9. 点击 `Run Suite` 运行整套自动化测试。
10. 在结果区查看每条请求的执行状态、断言结果和所属场景。

适合的使用时机：

- 提测前做一次全链路验证
- 每日巡检时跑固定检查项
- 变更后快速回归核心接口
- 发布前执行一套稳定的回归清单

## 9.3 运行结果怎么看

套件执行完成后，可以重点关注以下信息：

- 顶部会显示当前运行的 `Suite`
- 每条请求下方会显示 `Scenario`
- 状态图标可快速判断通过、失败或跳过
- 点击状态码可以查看该次执行的响应详情
- `Run Again` 会按最近一次套件执行计划再次运行

建议的排查顺序：

1. 先看哪个场景失败。
2. 再看该场景下是哪个请求失败。
3. 打开响应详情看状态码、断言和错误信息。
4. 修正后重新运行单个场景或整套套件。

## 10. 套件执行规则

场景套件执行时遵循以下规则：

- 同一个接口如果在多个场景中出现，会按场景顺序重复执行，不会被自动去重。
- 每个场景会沿用自己保存时的请求顺序。
- 每个场景会沿用自己保存时的请求延迟 `Delay`。
- 套件是“场景的组合器”，不会覆盖单个场景内部的请求选择和顺序。

## 10.1 自动化测试编排建议

为了让套件更适合长期复用，建议遵循以下约定：

- 一个场景只做一类事情，不要把过多步骤堆进同一个场景。
- 把高频复用步骤拆成独立场景，例如登录、鉴权、清理数据。
- 套件名称直接体现用途，例如 `Smoke`、`Regression`、`Release Check`。
- 尽量保持场景顺序稳定，避免频繁改动同一套件的执行语义。
- 如果某个场景只在特殊时期才需要，建议单独做一个套件，不要直接塞进通用回归套件。

## 11. 删除联动规则

- 删除某个场景后，所有引用该场景的套件会自动移除对应引用。
- 如果某个套件在清理后已经没有任何场景，会被自动删除。
- 删除套件不会影响原有场景。

## 12. Reset 与 Run Again 行为

- 未选中场景时，`Reset` 会恢复当前集合默认的 Runner 配置状态。
- 选中场景时，`Reset` 会恢复到该场景最近一次保存时的配置。
- 套件执行结束后，`Run Again` 会按上一次套件执行计划再次运行。

## 13. 当前支持范围

- 支持当前 Collection 中已保存的请求。
- 支持保存请求勾选结果与执行顺序。
- 支持标签过滤。
- 支持请求延迟。
- 支持多个场景组合执行。
- `gRPC` 请求在 Runner 中会被跳过。
- 临时请求 `transient request` 不会被纳入场景或套件。

## 14. 数据持久化位置

场景与套件都跟随集合配置保存：

- `bru` 格式集合：保存在 `brunoConfig`
- `yml / opencollection` 格式集合：保存在 `opencollection.yml` 的 Bruno 扩展字段

当前使用的字段：

- `brunoConfig.runnerScenarios`
- `brunoConfig.runnerScenarioSuites`

在 `opencollection.yml` 中，对应 Bruno 扩展字段为：

- `extensions.bruno.runner.scenarios`
- `extensions.bruno.runner.suites`
